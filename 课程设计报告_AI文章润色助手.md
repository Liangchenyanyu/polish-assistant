## 一、项目需求分析

### 1.1 项目背景

文章润色是写作流程中的重要环节，涉及语法修正、风格优化、逻辑梳理、降重改写、去 AI 痕迹等多个维度。传统人工润色效率低、成本高，且受限于润色者的专业水平。随着大语言模型（LLM）技术的快速发展，利用大模型强大的文本理解与生成能力来辅助写作已成为可能。

本课程设计旨在构建一个基于大模型的**智能文章润色系统**，面向学生群体（论文降重、报告润色）、职场人士（邮件润色、方案优化）以及自媒体创作者（去 AI 痕迹、通俗化写作），提供从基础润色到高级文本处理的完整工作流。

### 1.2 功能需求分析

| 编号 | 功能模块 | 具体需求 | 优先级 |
|:----:|----------|----------|:------:|
| F-01 | **智能润色核心** | 支持学术/商务/自媒体/简洁/自定义五种风格；全文/逐段两种模式；轻/中/深三级强度控制 | 高 |
| F-02 | **六大润色工具** | 智能润色、降重改写、去AI痕迹、通俗化、AI续写、话术转换（5种子类型） | 高 |
| F-03 | **角色定制系统** | 资深编辑、语文老师、商务顾问、学术导师四种角色人格 | 高 |

## 四、智能润色与流式对话（功能1简介）

### 4.1 功能概述

智能润色是本系统的**核心功能**。用户输入待润色的文本后，系统通过如下流程完成润色：

```
用户输入文本 → 文件上传 / 直接粘贴
       ↓
   参数配置（工具/风格/角色/强度/字数）
       ↓
   大模型润色处理（流式 SSE）
       ↓
   结果展示（对话气泡 + 实时渲染）
       ↓
   导出 / 继续优化 / 智能分析
```

### 4.2 核心流程

**Step 1 — 输入文本**：用户可在输入框中直接粘贴文本，或点击上传按钮上传 `.txt` / `.md` 文件（最大支持数万字的长文）。文件上传后显示文件卡片（文件名 + 类型 + 大小），支持一键清除。

**Step 2 — 配置参数**：在底部工具栏中设置：
- **润色工具**：智能润色/降重改写/去AI痕迹/通俗化/AI续写/话术转换
- **润色风格**：学术严谨/商务正式/自媒体活泼/简洁凝练/自定义
- **角色选择**：资深编辑/语文老师/商务顾问/学术导师
- **润色强度**：轻度（仅修错）/ 中度（优化语句）/ 深度（重构全文）
- **目标字数**：0-5000 字滑块控制

**Step 3 — 流式生成**：点击发送按钮后，系统通过 **SSE（Server-Sent Events）** 将大模型生成的润色结果**逐字推送到前端**，用户可实时看到文字逐字出现的效果，并显示闪烁光标指示生成状态。生成过程中可随时点击**红色停止按钮**中断。

**Step 4 — 结果展示**：润色结果以聊天气泡形式展示——用户消息蓝色右对齐，AI 回复白色左对齐，支持 Markdown 格式渲染（代码块、表格、列表、引用块等）。

### 4.3 多轮对话优化

系统支持**上下文连贯的多轮追问**。用户可以在第一次润色后，继续发送指令（如「再简洁一些」「加入更多数据支撑」），系统会基于之前的对话历史进行优化，保持前后文的连贯性。

### 4.4 前后对比

润色完成后，用户可在「对比」Tab 中查看**原文 vs 润色后**的双栏对比视图：
- 左栏：原文（橙色底色）
- 右栏：润色后（绿色底色）
- 支持同步滚动和独立滚动两种模式

### 4.5 技术实现要点

- **前端**：`usePolish.ts` Hook 使用 `fetch + ReadableStream` 解析 SSE 事件流，逐 token 累加文本并触发 React 重渲染
- **后端**：`polish_engine.py` 编排完整流程：RAG 检索 → COSTAR 提示词构建 → 大模型 API 流式调用 → SSE 事件推送
- **对话持久化**：所有对话自动保存到 `localStorage`（key: `polish-assistant-conversations`），刷新/重启不丢失

## 五、六大工具与角色定制系统（功能2简介）

### 5.1 六大润色工具

系统提供六大专用润色工具，每种工具拥有独立的 COSTAR 提示词，覆盖不同写作场景：

| 工具 | 图标 | 核心功能 | 提示词策略 |
|------|:----:|----------|------------|
| **智能润色** | ✨ | 三级强度控制（轻度仅修错 / 中度优化语句 / 深度重构全文） | 强度 + 风格 + 5条负向约束 |
| **降重改写** | 🔄 | 同义替换 + 句式重构（主动↔被动、长句拆分、短句合并）+ 语序调整 | 「改形不改意」原则 |
| **去AI痕迹** | 👤 | 消除机器化句式、过度工整、空洞修饰、缺乏个性、句式单一五大特征 | 模拟真人手写风格 |
| **通俗化** | 📖 | 化繁为简 + 举例 + 拆长句 + 加解释 + 降密度 | 专业内容 → 大众易懂 |
| **AI续写** | ➡️ | 基于前文风格自然续写，保持风格/逻辑/内容一致 | 字数控制 + 分界线标记 |
| **话术转换** | 🎭 | 5种子类型：委婉表达 / 感染力增强 / 积极转化 / 正式化 / 口语化 | 保留核心诉求 + 效果对比 |

### 5.2 四种角色人格

系统内置四种角色 persona，每种角色以不同的专业身份和语气完成润色：

| 角色 | 图标 | 人格定义 | 适用场景 |
|------|:----:|----------|----------|
| **资深编辑** | ✏️ | 15年出版编辑经验，注重专业性、可读性和传播效果 | 通用场景 |
| **语文老师** | 🍎 | 20年教龄，耐心细致，如春风化雨般的温和点评 | 学生写作、基础语法 |
| **商务顾问** | 💼 | 500强企业商务沟通培训师，目标导向 + 行动号召 | 商务邮件、方案汇报 |
| **学术导师** | 🎓 | 大学教授，指导过数百篇论文，严谨规范 | 学术论文、研究报告 |

## 六、知识库增强检索RAG（功能3简介）

### 6.1 功能概述

知识库增强检索（RAG, Retrieval-Augmented Generation）是本系统的重要创新点。在用户提交润色请求时，系统会从写作规范知识库中检索最相关的规范条文，将其作为上下文注入大模型的提示词中，使润色结果更具专业性和规范性。

### 6.2 双引擎检索策略（核心创新）

由于 HuggingFace 模型下载在部分网络环境下不可达，系统创新性地实现了**双引擎检索策略**，确保知识库在任何网络环境下 100% 可用：

```
用户查询
    │
    ▼
┌─────────────────────┐
│ 主引擎：ChromaDB     │  网络可达？
│ + HuggingFace Embed  │──── Yes ──→ 向量相似度检索 (Top-K=3)
│ (bge-small-zh-v1.5) │                │
└─────────────────────┘              返回结果
    │ No（网络不可达）
    ▼
┌─────────────────────┐
│ 回退引擎：TF-IDF     │  始终可用（零网络依赖）
│ + BM25 关键词检索    │──────→ 关键词匹配检索 (Top-K=3)
│ (纯 Python 实现)     │         │
└─────────────────────┘       返回结果
```

**回退引擎实现要点：**
- 使用 `sklearn.feature_extraction.text.TfidfVectorizer` 构建索引
- BM25 算法计算查询与文档的相关度评分
- 启动时立即构建 TF-IDF 索引，标记 `_initialized = True`
- 确保知识库在任何网络环境下 100% 可用

### 6.3 RAG 完整流程

```
用户输入文章 + 润色参数
         │
         ▼
  构建检索查询（根据 style 拼接查询词）
         │
         ▼
  ┌──────────────────────────────┐
  │  文本切分                      │
  │  RecursiveCharacterTextSplitter│
  │  Chunk Size: 500, Overlap: 50  │
  │  分隔符: \n\n → \n → 。→ ；  │
  └──────────────────────────────┘
         │
         ▼
  ┌──────────────────────────────┐
  │  向量化 / TF-IDF               │
  │  bge-small-zh → 768维向量     │
  │  或 TF-IDF 稀疏向量（回退）    │
  └──────────────────────────────┘
         │
         ▼
  ┌──────────────────────────────┐
  │  相似度检索 (Top-K=3)          │
  │  返回最相关的 3 个规范片段      │
  └──────────────────────────────┘
         │
         ▼
  上下文拼接 → 注入 System Prompt
         │
         ▼
  大模型生成润色结果（含规范依据）
         │
         ▼
  SSE result 事件透传 kb_refs 到前端
         │
         ▼
  前端「引用来源」Tab 展示
```

用户可随时通过前端 `CorpusModal` 弹窗管理知识库语料：

## 七、智能文本分析系统（功能4简介）

### 7.1 功能概述

智能文本分析系统是一个**纯 Python 实现、零网络依赖**的文本质量评估工具，可从六个维度全面分析用户输入的文本质量。用户可通过顶部 Header 的 📊 图表按钮打开分析弹窗。

### 7.2 六大分析维度

| 分析维度 | 实现方法 | 输出内容 |
|----------|----------|----------|
| **可读性评分** | 改编中文可读性公式（句长 + 罕见字比例 + 标点密度） | 0-100 分 + 难度等级 + 适合年级 + 6项统计指标 |
| **语病检测** | 12种中文语病正则模式匹配 | 错误描述 + 上下文 + 修改建议 |
| **重复度检测** | 4-gram 短语重复分析 | 重复短语 + 出现次数 + 位置示例 |


| 30-49 | 较难 | 大学 | 专业文章 |
| 0-29 | 困难 | 研究生 | 学术论文 |

### 7.4 语病检测覆盖

```python
from fpdf import FPDF
```

### 9.2 遇到的困难与解决方案

**困难一：HuggingFace 模型下载失败导致知识库不可用**

- **问题描述**：ChromaDB 依赖 HuggingFace 的 `bge-small-zh-v1.5` 嵌入模型，但在校园网络环境下 `huggingface.co` 及镜像站 `hf-mirror.com` 均连接超时，导致向量数据库初始化失败，知识库检索功能完全不可用。
- **解决方案**：设计**双引擎检索策略**——主引擎保持 ChromaDB + HuggingFace Embedding，同时实现 TF-IDF + BM25 纯 Python 回退引擎。启动时立即构建 TF-IDF 索引并标记状态为已就绪，知识库初始化超时不再抛异常，自动降级为 TF-IDF 检索。
- **改进效果**：知识库在任何网络环境下 100% 可用，引用来源 Tab 正常显示文档名和相关度评分。

**困难二：前端引用来源始终显示 0**

- **问题描述**：虽然后端知识库检索能返回结果，但前端「引用来源」Tab 始终显示引用数量为 0。
- **根因分析**：后端 `polish_engine.py` 的 `polish_article()` 方法在检索到知识库片段后，未通过 SSE `result` 事件将 `kb_refs` 数据透传到前端。

**困难三：弹出层被父容器 overflow:hidden 裁剪**

- **问题描述**：前端工具选择器、更多设置等 Popover 弹出层在展开时常被祖先元素的 `overflow: hidden` 样式裁剪，导致部分选项不可见。
- **解决方案**：实现 Portal 弹出层组件，使用 `ReactDOM.createPortal` 将弹出层挂载到 `document.body`，脱离父容器的 CSS 上下文，独立计算展示位置。

**困难四：学术风格润色结果过于死板**

- **问题描述**：所有风格共用默认 Temperature 0.7，但学术场景需要对原文保持更高的忠实度，不宜有过多创造性改写。
- **解决方案**：按风格设置差异化默认 Temperature——学术风格 `0.5`（严谨）、自媒体风格 `0.9`（活泼），其他风格保持 `0.7`。

### 9.3 心得体会

通过本次课程设计，我深入理解了大模型应用开发的完整流程：

1. **提示词工程的重要性**：COSTAR 框架让提示词从「随意描述」变为「工程化设计」。V1 到 V3 的迭代让我深刻体会到，一个好的提示词模板需要经过多次「设计→测试→优化」的循环。负向提示词对大模型幻觉的抑制效果显著。

2. **工程鲁棒性不可忽视**：HuggingFace 不可达的问题让我认识到，依赖外部服务的系统必须有**降级方案**。双引擎检索策略的设计思想——「主路优先，备路兜底」——值得在其他模块中推广。

3. **前后端数据流的一致性**：SSE 事件流的每个环节（`token → progress → result → paragraph_done → done`）都需前后端精确对齐，一个字段的遗漏就可能导致功能异常。

4. **用户体验的细节打磨**：流式光标、Portal 弹出层、双栏对比视图、引用来源卡片等交互细节，虽然技术实现不复杂，但对用户体验的提升是决定性的。
